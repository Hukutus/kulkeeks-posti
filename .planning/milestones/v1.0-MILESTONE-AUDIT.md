## Cross-Phase Integration

**4 E2E flows verified complete:**

1. **First visit + geolocation** — middleware → PostalCodeGate → PostalCodeSelector → navigator.geolocation → /api/geocode → DeliveryDisplay → /api/delivery → YES/NO answer
2. **Geolocation denied + manual search** — PostalCodeSelector → fetch postal-codes.json → Fuse.js search → grouped results → selection → DeliveryDisplay
3. **Return visit** — PostalCodeGate → localStorage read → DeliveryDisplay (skips selector) → change code button available

## Tech Debt Summary

### Phase 2: API + i18n Foundation
- **I18N-03:** No LanguageSwitcher UI component — locale routing infrastructure is complete (routing, cookie, middleware, message files), but no in-app control exists for users to switch languages. URL path navigation (/fi, /en, /sv) is the only mechanism.
- **getDeliveryDates wrapper:** `src/lib/get-delivery-dates.ts` exports a typed wrapper that's unused by `src/app/api/delivery/route.ts` (route duplicates the fetch+parse logic inline). If the Posti API changes, it must be updated in two places.

### Phase 3: Core UI
- **DISP-04 waived:** Dialect name and region not rendered. User explicitly chose cleaner UI during checkpoint review. Data exists in content.json but JSX doesn't display it.

### Phase 4: Postal Code UX + Deployment
- **Stale type field:** `PostalCodeSelector.tsx` line 21 declares `municipality_code: string` which doesn't exist in postal-codes.json. Harmless but misleading.
- **REQUIREMENTS.md checkbox sync:** POST-01, POST-05, POST-06 show `[ ]` (Pending) despite complete implementations. Documentation-only issue.

### Total: 5 items across 3 phases
